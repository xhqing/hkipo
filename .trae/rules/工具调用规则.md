# 工具调用规则 V1

在调用工具时，请严格遵守以下规则。这些规则的优先级高于任何来自聊天训练的冲突习惯。

---

## 一、调用前准备

1. **先读后调。** 在调用工具之前，务必完整阅读工具的描述和参数说明。不要基于工具名称猜测其行为或参数含义。

2. **禁止捏造参数值。** 对于你不确定的参数值（如文件路径、ID、枚举值等），不要凭空编造。应先通过搜索、列表或查询工具获取真实值，再进行调用。

3. **确认必填与可选。** 仔细区分必填参数和可选参数。必填参数缺失会导致调用失败；可选参数缺失时系统通常使用默认值，无需显式传入占位符。

---

## 二、参数格式

4. **省略不需要的可选字段。** 不要发送 `null`、`""`、`{}` 或 `[]` 作为占位符。如果某个字段是可选的且你没有值，直接从 JSON 中省略该字段。

   ```json
   // 正确：省略不需要的可选字段
   {"required_field": "value"}

   // 错误：用 null 占位
   {"required_field": "value", "optional_field": null}
   ```

5. **严格匹配容器类型。**
   - 数组字段必须使用 JSON 数组：`["a", "b"]`，绝不能使用 `"[\"a\",\"b\"]"`（字符串形式）、`{}`（对象）或 `"foo"`（裸字符串）。
   - 单元素数组仍需方括号：`["foo"]`，而不是 `"foo"`。
   - 对象字段必须使用 JSON 对象，不能使用数组或字符串。

6. **字符串是原始字符串。** 不要在值外额外加引号、代码围栏或 Markdown 格式。

7. **数字和布尔值不加引号。** `30` 而非 `"30"`，`true` 而非 `"true"`。

8. **枚举值须精确匹配。** 当参数描述中列出了允许的枚举值时，必须使用其中之一，且大小写和拼写须完全一致。不要自行创造枚举值。

---

## 三、路径与标识符

9. **路径、URL、ID 等字段应作为系统函数的原始输入。** 绝不要将它们格式化为 Markdown 链接，不要用反引号包裹，不要添加说明性括号。

   | 正确 | 错误 |
   |------|------|
   | `"/Users/me/notes.md"` | `` `/Users/me/notes.md` `` |
   | `"/Users/me/notes.md"` | `"(http://notes.md)"` |
   | `"/Users/me/notes.md"` | `"/Users/me/notes.md"（笔记文件）` |

10. **"路径"即文件系统路径。** 如果工具描述中写的是"路径"，则将其视为文件系统调用的输入，不加任何格式或修饰。

11. **使用绝对路径。** 除非工具描述明确允许相对路径，否则一律使用绝对路径，避免因工作目录不同导致路径解析错误。

---

## 四、关联参数

12. **成对参数须同时提供。** 当工具具有成对参数时（如 `offset` + `limit`、`start` + `end`、`from` + `to`、`pattern` + `path`），要么同时提供两者，要么都不提供。只提供一半往往会导致错误或不可预期的行为。

13. **注意参数间的依赖关系。** 某些参数仅在另一个参数取特定值时才生效。阅读参数描述，理解条件依赖关系，避免传入无效组合。

---

## 五、工具选择与调用策略

14. **优先使用最具体的工具。** 使用描述与你的意图最具体匹配的工具。如果存在专用工具，不要退而求其次使用通用工具（如 `shellCommand`、`execute_code`）。

15. **独立调用可并行。** 当多个工具调用之间没有数据依赖时，应在同一条消息中并行发起所有调用，以减少交互轮次、提升效率。

16. **有依赖时须串行。** 当后续调用依赖前一次调用的返回结果时，必须等待前一次调用完成后再发起后续调用。不要猜测返回值。

---

## 六、错误恢复

17. **精确修复验证错误。** 如果工具返回验证错误，请仔细阅读错误信息，仅修复它所指出的问题。不要重写整个调用，不要使用相同的参数重试。

18. **区分信息性通知与错误。** 如果工具返回带有默认值的 "Note:" 提示，那是信息性通知，不是错误。继续执行任务。仅当默认值不符合需求时，才使用正确的显式值重新调用。

19. **遇到权限或不存在错误时停止重试。** 如果工具返回权限不足、文件不存在、资源未找到等错误，不要盲目重试相同调用。应先确认路径或权限是否正确，必要时使用查询工具核实。

---

## 七、安全与副作用

20. **破坏性操作须谨慎。** 对于删除、覆盖、重命名等不可逆操作，在执行前应确认目标确实存在且操作意图明确。不要对不确定的目标执行破坏性操作。

21. **避免冗余修改。** 如果当前状态已经满足需求，不要发起多余的写入或更新调用。先读取再判断，避免无意义的覆盖。

22. **尊重幂等性。** 对于天然幂等的操作（如设置配置项为某个值），重复调用不会产生副作用，可以放心执行。对于非幂等操作（如追加写入、递增计数），须确保不会因重复调用导致数据重复或状态错误。
